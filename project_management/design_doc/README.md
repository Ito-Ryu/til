# Design Documents (設計書)

## 概要
### Design Doc（デザインドック）
新しい機能の追加やシステムの改修など、プロジェクトの変更点に関する詳細な設計内容を記したドキュメント。</br>
実装に着手する前の段階で、技術的なアプローチ、目的、非機能要件、代替案などを整理し、チーム内での合意形成を図ることを目的とする。

## Design Doc を作成する目的

* **実装前のレビュー**: 複雑な機能の設計を早期にレビューすることで、設計ミスや技術的な不確実性を排除する
* **技術的な合意形成**: 複数の実現方法から、なぜこの案が最適なのかを明文化し、関係者間で認識を合わせる
* **将来のメンテナンス性の向上**: 実装時の設計意図を詳細に残しておくことで、運用や将来的な改修時に貴重な情報源となる

## 構成要素の説明

詳細は `template.md` 

1. **概要 (Overview)**: 提案する変更の要約、一言で説明
2. **背景・目的 (Context & Goals)**: 開発が必要となった理由、解決したいビジネス/技術的な課題
3. **目標と非目標 (Goals & Non-Goals)**: 達成したいことと、あえて今回のスコープから外すことを明記
4. **提案設計 (Proposed Design)**: システム構成図、シーケンス図、データフローなど
5. **詳細設計 (Detailed Design)**: API 定義、DB スキーマ、具体的なアルゴリズム、モジュール構成
6. **検討した代替案 (Alternatives Considered)**: 検討したが採用しなかった他の案、不採用の理由（コスト、期間、技術的制約など）
7. **影響・リスク (Impact & Risks)**: パフォーマンス、セキュリティ、監視、運用面への影響
8. **実装プラン (Implementation Plan)**: フェーズ分けした作業工程、主要なマイルストーン
